35// The usage reason for splitting the Chunk allocator into a ChunkPool and a
36// ByteAllocator is so that similarly accessed objects get allocated next to each
37// other within the same Chunk, which is managed by the ByteAllocator. The
38// ChunkPool is basically just a resource pool. Not thread safe anything :-)
39// Locking is bad mkkk...
41
43// Usage example of setting up a ChunkPool, a ByteAllocator and allocating some
44// bytes, for the lulz.
46
47/*
48 OmniPvdChunkPool pool;
49
50 // set the individual Chunks to a size of 100k bytes
51 pool.setChunkSize(100000);
52
53 // all allocations must go through a ByteAllocator, so create one
54 OmniPvdChunkByteAllocator allocator;
55
56 // connect the ByteAllocator to the pool
57 allocator.setPool(&pool);
58
59 // we allocate 100 bytes through the ByteAllocator
60 unsigned char *someBytes=allocator.allocBytes(100);
61*/
62
64// Once we want to cleanup the allocations, return the Chunks to the ChunkPool,
65// which is also done silently in the destructor of the ByteAllocator.
67
68/*
69 allocator.freeChunks();
70*/
71
73// Memory management warning! The ByteAllocator does not deallocate the Chunks,
74// but must return the Chunks to the ChunkPool before going out of scope. The
75// ChunkPool can only deallocate the Chunks that is has in its free Chunks list,
76// so all ByteAllocators must either all go out of scope before the ChunkPool
77// goes out of scope or the ByteAllocators must all call the function freeChunks
78// before the ChunkPool goes out of scope.
